# 第14章 权限的传递：能否转交、能否复制
权限一旦被授予，能否被接收者再次转交给第三方？能否被复制多份？这两个问题决定了权限在系统中的流动方式，也直接影响安全模型的强度。不同的内核对此给出了不同的答案。

## 14.1 传递与复制的本质区别

权限传递（transfer）意味着原持有者将权限交给另一个主体，之后原持有者不再拥有该权限。权限复制（copy）则意味着原持有者保留权限的同时，另一个主体也获得一份相同的权限。

传递类似于"给出去"，权限的所有权发生转移；复制类似于"复印一份"，权限的持有者增加。两者的安全含义完全不同：传递缩小了权限的分布范围，复制则扩大。一个谨慎的系统通常限制复制的能力，因为每次复制都增加了权限被滥用的风险。

在能力系统的学术讨论中，两种传递方式被区分为法律上的（de jure）和事实上的（de facto）传递。前者指主体直接获得读取信息的权威；后者指主体通过其他方式获得了信息（通常以副本形式，借助他人帮助），但未必获得直接读取信息的权威。前者对应权限本身的转移，后者对应数据本身的流动。

## 14.2 能力系统中的传递与派生

需要说明的是，"能力"这个词在学术界有两种用法：一种是指不可伪造的资源访问令牌（如seL4、HIC），另一种是指Linux系统中拆分root权限的"特权单元"（如CAP_NET_ADMIN）。本书讨论的能力系统指前一种。

在基于能力的系统中，权限的流动通过两种基本操作实现：复制（Copy）和派生（Mint），二者的安全含义有显著区别。

复制操作将一个能力从源槽复制到目标槽，复制后源槽的能力仍然存在，目标槽获得一个相同权限的副本。这意味着权限的持有者数量增加。

派生操作（在seL4中称为Mint）从一个现有能力创建一个新能力，但新能力是原能力的子集——可以在派生过程中移除某些权限（例如，将读写能力降级为只读能力）。派生后，原主体仍然持有原能力，新能力归另一个主体所有。

seL4对这两种操作的处理在能力派生树中体现为不同的树结构：复制操作在派生树上创建一个原能力的直接子节点，而派生操作则创建一个受限权限的新分支。这种区别使得审计人员能够追踪每个能力的来源和权限演变历史。seL4的这种句柄繁殖控制模型被称为基于take-grant的模型，可以递归地撤销句柄的所有继承权限。

## 14.3 ACL与标签系统中的传递

在基于ACL的系统中，权限的"传递"通常通过修改访问控制列表来实现。一个主体如果拥有对某个客体的管理权限（例如文件的所有者权限），就可以修改该客体的ACL，将其他主体加入列表。这本质上是一种权限的派生：所有者将其部分权限"复制"给了其他主体，而所有者自身不失去任何权限。

ACL模型的这种特性使得权限扩散容易发生。一个文件的所有者可以将读权限授予任意多个用户，每个获得权限的用户又可以（如果被允许）继续授予其他人。追踪权限的传播路径变得困难，这给安全审计带来了挑战。

基于标签的系统则采用不同的方式。权限不是通过修改ACL或传递能力来授予，而是通过标签匹配规则。一个主体的标签决定了它可以访问哪些标签的客体。要"传递"权限，通常需要改变主体的标签或修改策略规则。这种设计限制了普通用户自主传播权限的能力，将权限控制集中到策略管理者手中。

## 14.4 复制与权限扩散的风险

权限复制带来的主要风险是权限扩散（permission proliferation）。一个权限被复制一次，就可能被再次复制，最终传播到原本不该拥有它的主体手中。攻击者可以利用合法的复制机制，逐步扩大自己的权限范围。

对象能力模型（Object-Capability Model）通过一条根本原则来对抗这种风险：只有连接才能产生连接。一个对象只有通过一条先前存在的引用链才能获得另一个对象的引用。这意味着，如果程序A从未获得过对资源R的能力，那么它无法通过任何方式"制造"出这个能力——能力的来源必须是一条可追溯的授权链。这条原则从根源上限制了权限的任意扩散。

为了控制复制风险，系统还可以采用以下手段：限制只有特定权限才能进行复制（例如，只有持有"复制"权能的主体才能派生新能力）；限制复制后的权限不能再次复制（单次授权）；或者要求复制操作必须经过审计。

## 14.5 Linux中的能力继承与传递

在Linux的能力机制中，权限的传递规则因操作类型而异。当进程通过fork创建子进程时，子进程获得父进程能力集的精确副本——这是一个典型的复制操作，父子进程拥有相同的能力集。当进程通过exec执行新程序时，情况不同：新程序的能力由文件能力（file capabilities）和进程原有能力共同计算决定，而非简单继承父进程的能力。此外，默认情况下能力不会被子进程继承——如果shell拥有大量能力，其子进程（如gcc或gmake）并不会自动获得这些能力，除非这些可执行文件本身被赋予了文件能力。这种设计在便利性和最小权限之间取得了一种平衡。

## 14.6 seL4 的能力传递：Move、Copy 与 Mint

seL4 微内核将权限传递建模为能力空间（CSpace）中的操作，三种原语覆盖了全部传递语义。

Move（移动）是最严格的传递方式。调用 seL4_CNode_Move() 将能力从一个能力槽移动到另一个能力槽。移动后，源能力槽变为空——能力在系统中只有一个副本。这种设计限制了权限的扩散：一个能力被移动后，原持有者不再拥有该能力。Move 操作不增加能力引用计数，撤销时不需要扫描多个副本。

Copy（复制）创建能力的精确副本。调用 seL4_CNode_Copy() 在目标能力槽中创建源能力的副本。复制后，源和目标拥有完全相同的权限。这种设计允许能力的并行使用，但代价是增加撤销的复杂度——撤销操作需要递归地扫描所有派生副本。seL4 通过能力派生树（双向链表）追踪这些副本关系，确保撤销可以遍历整个子树。

Mint（铸造）是带衰减的复制。调用 seL4_CNode_Mint() 在目标槽中创建一个新能力，但在创建过程中可以限制新能力的权限。Mint 是 seL4 实现最小权限原则的主要手段——父能力可以派生出只读的子能力，子能力可以继续派生只读的子能力，但任何能力都不能派生权限超出自身范围的新能力。权限单调递减是 seL4 能力系统的核心不变量。

三种原语组合起来覆盖了所有传递场景：Move 适合能力所有权转移，Copy 适合权限共享，Mint 适合权限衰减委托。seL4 不提供全局能力表——每个线程有自己的能力空间（CNode 树），能力的传递本质上是不同能力空间之间的槽操作。这与 HIC 的全局 ID 表模型形成对比：seL4 的传递是空间间的操作，HIC 的传递是统一 ID 表的归属变更。

## 14.7 HIC中的全局能力表传递

HIC采用能力模型，支持权限的传递和派生。Core-0维护全局能力表，Privileged-1服务之间可以通过能力传递实现权限的委托，通过能力派生实现权限的衰减。与seL4类似，HIC的能力派生操作允许在传递过程中减少权限（例如，将一个具备读写能力的能力派生出只读版本后交给下游服务），从而贯彻最小权限原则。

## 14.8 本章小结

权限的传递与复制是访问控制系统中不可回避的问题。传递意味着权限所有权的转移，复制意味着权限持有者的增加。复制带来的权限扩散风险是安全设计的主要挑战之一。在基于能力的系统中，seL4 用 Move（移动）、Copy（复制）、Mint（铸造衰减副本）三种原语覆盖了完整的传递语义，能力空间之间的操作独立于全局表。对象能力模型通过"只有连接才能产生连接"原则从根源上限制权限的任意扩散。ACL系统的权限传递往往更灵活但更难追踪。不同系统在设计传递规则时需要在灵活性和可控性之间做出权衡，而能力派生机制和最小权限原则是控制扩散风险的关键手段。
