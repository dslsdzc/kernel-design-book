# 第77章 架构差异对调度模型的影响：多核同步、内存屏障、原子操作
第22至37章讨论了调度模型的核心问题——公平、效率、优先级、中断等。这些讨论隐含了一个假设：调度器运行在单核或多核一致共享内存的环境中。然而，不同处理器架构在 **多核同步**、**内存访问顺序** 和 **原子操作** 支持上存在根本性差异。这些差异直接影响调度器的设计：调度器核心数据结构（如就绪队列）的并发访问如何保护；调度决策中涉及的共享变量如何保证顺序一致性；跨核的负载均衡和任务迁移如何正确同步；调度器如何在不牺牲性能的前提下保证正确性。本章从架构差异的视角重新审视调度模型，讨论x86、ARM、RISC-V在内存模型、原子操作和同步原语上的差异，以及这些差异对调度器设计的影响。

## 77.1 内存一致性模型：从强到弱的频谱

内存一致性模型定义了多核系统中一个核的内存访问操作何时对其他核可见。不同架构对内存访问顺序的保证强度不同，形成了从强一致性到弱一致性的频谱。

x86-TSO是目前最强的主流内存模型。TSO代表总存储顺序（Total Store Order），它保证所有核看到的内存写入顺序与程序顺序一致。x86保证以下特性：写操作不会与其他写操作重排；写操作不会与之前的读操作重排；读操作不会与之前的读操作重排（但在某些情况下读操作可能与之前的写操作重排，即Store-Load重排）。这意味着在多核x86上编写并发代码时，开发者不需要在所有访问路径上都插入内存屏障。x86为强内存模型，ARM和RISC-V等弱内存架构需要显式屏障。x86-TSO模型已经被形式化定义，并在HOL4定理证明器中被验证。

ARMv8采用弱内存模型，允许更多的重排优化以提高性能。ARMv8允许多种重排：读操作可以重排到读操作之前和之后；读操作可以重排到写操作之前（但写操作不会重排到读操作之前）。这种弱模型使ARM处理器在低功耗嵌入式设备上实现了更好的能效，但代价是需要更多的显式内存屏障来保证正确的并发行为。ARM的Load-Acquire（LDAR）和Store-Release（STLR）指令提供了获取-释放语义，分别用于确保后续操作不会重排到加载之前，以及确保所有之前的操作不会重排到存储之后。

RISC-V RVWMO（RISC-V Weak Memory Ordering）是一种极其宽松的内存模型。在RVWMO中，除非使用显式的FENCE指令或获取-释放原子操作，否则不同hart之间的内存访问顺序几乎不受约束。RVWMO允许读-读重排、读-写重排、写-写重排。此外，RVWMO还包含非多拷贝原子性（Non-multicopy atomicity）——一个hart的存储操作不是同时对所有其他hart可见的。CSR（控制和状态寄存器）访问也是弱顺序的，其他hart或设备可能以与程序顺序不同的顺序观察到CSR访问。

三种架构的内存模型强度差异对调度器设计有直接而深远的影响。

## 77.2 原子操作：不同架构的硬件原语

原子操作是多核同步的基础。不同架构支持的原子操作原语不同，这直接影响调度器中无锁数据结构的实现。

x86提供Compare-And-Swap（CMPXCHG）指令系列。CMPXCHG原子性地比较内存位置的值与期望值，如果相等则替换为新值，并返回原值。x86的LOCK前缀可以将任何内存操作（ADD、SUB、AND、OR、XCHG等）变为原子操作。LOCK前缀通过锁定总线或缓存行实现原子性，并自动充当完整的内存屏障（LFENCE、SFENCE、MFENCE），保证所有核看到一致的执行顺序。x86指令集的CMPXCHG等原子指令，相比RISC-V的LR/SC等原子指令，为调度器的无锁实现提供了更直接的硬件支持。

ARM采用Load-Reserved/Store-Conditional（LDREX/STREX）机制。LDREX从内存地址加载值并标记该地址为独占访问；STREX尝试将新值存储到同一地址，只有当该地址未被其他核修改时，存储才会成功并返回0；否则返回1表示失败，需要重试。ARMv8.1引入了原子操作指令（CAS、SWP等），但LDREX/STREX仍广泛用于实现各种无锁数据结构。ARM的弱内存模型要求开发者使用显式的数据同步屏障（DSB）和数据内存屏障（DMB）来保证内存访问顺序。

RISC-V采用Load-Reserved/Store-Conditional（LR/SC）机制，与ARM类似但设计更加精简。LR指令保留对内存地址的独占访问；SC指令在有保留的情况下条件性地写入。RISC-V的LR/SC需要配合FENCE指令来保证内存顺序。RVWMO的弱内存模型意味着LR/SC本身不提供顺序保证，开发者必须在关键位置插入FENCE。RISC-V原子扩展中还提供了AMO（Atomic Memory Operation）指令，支持原子加法、交换、位操作等。RVWMO中FENCE指令的顺序约束由前序集合和后序集合参数决定，支持从读-读到写-写等多种顺序组合。

从调度器设计角度看，x86的CMPXCHG配合LOCK前缀提供了最强的原子操作语义，实现无锁数据结构时需要的屏障最少。ARM和RISC-V的LR/SC机制需要处理重试逻辑，且在弱内存模型下需要仔细放置屏障。RISC-V的RVWMO尤其严格，对调度器设计者的并发编程能力要求更高。

## 77.3 从x86迁移到弱内存架构的隐藏问题

随着ARM服务器和RISC-V设备的普及，原本为x86编写的内核代码被移植到弱内存架构时，可能暴露隐藏的并发错误。为x86-TSO模型编写和测试的程序在迁移到ARM或RISC-V时，可能表现出在x86上从未出现的行为。

在调度器上下文中，经典的危险模式是“双检查锁”模式用于初始化调度器数据结构。在x86上，由于TSO模型的强保证，这个模式可能侥幸工作；但在ARM或RISC-V上，由于读操作可能重排到写操作之前，其他核可能看到未完全初始化的结构。另一个经典场景是调度器队列操作。一个核在将任务加入就绪队列时，更新队列的tail指针；另一个核同时通过head指针读取队列。在x86上，由于存储-存储顺序被保证，只要tail在head之后更新，读取顺序就是正确的。但在ARM或RISC-V上，tail的更新可能被重排到head的更新之前，导致读核看到空的队列。

Linux内核通过Linux内核内存一致性模型（LKMM）来应对这一挑战。LKMM形式化了在弱内存架构上编写正确并发代码所需的约束。LKMM已被形式化验证，并在tools/memory-model/Documentation/explanation.txt中有详细说明。Paul McKenney曾提议引入新的smp_tmb()屏障，映射到x86的barrier()、ARM的stlr和PowerPC的lwsync，为调度器等核心子系统提供跨架构的一致性保证。LKMM的rmw-sequences机制确保了原子读-修改-写操作提供预期的顺序保证，但研究表明原子操作的内存顺序属性并非单调的——具有完整屏障语义的原子操作并不总是提供比释放屏障语义更强的顺序。这提示调度器设计者需要理解所使用的同步原语在不同架构上的精确语义。

## 77.4 调度器中的无锁同步与屏障优化

调度器是内核中最频繁执行的子系统之一，锁竞争会严重影响系统性能。因此，调度器大量使用无锁数据结构和细粒度同步来减少锁开销。这些无锁实现高度依赖原子操作和内存屏障，在不同架构上需要不同的实现策略。

就绪队列的无锁操作是现代调度器的核心优化。在Linux CFS调度器中，每个CPU核心拥有本地就绪队列（rq->cfs_rq），减少全局锁竞争。但负载均衡需要跨核心访问就绪队列，这时需要使用无锁读操作。RCU被广泛用于保护对就绪队列的读访问。rcu_dereference()和rcu_assign_pointer()通过隐式内存屏障保证了跨架构的顺序正确性。RCU允许读者无锁访问队列，写者更新队列时延迟释放旧版本，直到所有读者完成。

任务状态切换的内存顺序是调度器的关键路径。当调度器选择下一个任务执行时，需要更新rq->curr指针指向新任务的task_struct。这个指针的更新必须保证其他核看到的一致性：对任务状态的写入必须在rq->curr更新之前被其他核看到。rcu_assign_pointer()提供了必要的内存屏障来顺序化对task_struct的赋值和对rq->curr的赋值。在x86上，rcu_assign_pointer()通常只是普通赋值，因为TSO保证了顺序；但在ARM和RISC-V上，它会扩展为显式的存储释放屏障。由于rq->curr的更新是调度器的热路径，额外的屏障在ARM上可能引入约10-20纳秒的开销，而在RISC-V上可能更高。

无锁调度队列是学术界的研究方向。在众核处理器上开发无锁调度器面临共享资源状态和事务性内存机制等挑战。研究者提出了一种新的方法，使用无锁乐观并发控制来满足可扩展性的需求，并评估了不同调度器设计之间的干扰程度。LRSCwait方案通过允许争用的核心在等待之前核心完成原子访问时进入睡眠状态，消除轮询开销，为众核系统上的调度器同步提供了新的方向。这些研究表明，在弱内存模型上实现无锁调度器同步需要更精细的设计。

## 77.5 跨核同步：从锁到无锁的演进

调度域负载均衡是调度器中最复杂的跨核同步问题。在Linux中，负载均衡器定期扫描所有CPU核心的就绪队列，将任务从高负载核心迁移到低负载核心。负载均衡器使用调度域锁（rq->lock）来保护对远程队列的访问，但在大规模系统上，锁竞争可能成为瓶颈。为了解决这一问题，调度器在负载均衡路径中使用RCU保护对调度域结构的读访问，减少锁的使用。

跨核任务迁移需要精细的同步。当一个任务从源核心迁移到目标核心时，源核心需要将任务从自己的就绪队列中移除，目标核心需要将其添加到自己的队列中。这一过程涉及多个共享变量的更新，必须保证原子性。在x86上，LOCK前缀的原子操作可以完成这一任务；在ARM和RISC-V上，需要LDREX/STREX或LR/SC循环来实现。跨核迁移还需要使用核间中断（IPI）来通知目标核心有新任务到达。IPI本身是一个跨核同步操作，在不同架构上实现不同，但都依赖原子操作和内存屏障来保证IPI消息的传递正确性。

## 77.6 HIC中的调度同步

HIC在调度同步上采取了一种根本不同的策略——无锁设计。Core-0在单核执行时禁用中断来保证原子性，避免了传统锁机制带来的依赖关系和死锁风险。这种设计本质上绕开了多核同步的大多数问题，因为调度决策路径上不存在并发访问。

Core-0的原子性保证。HIC的Core-0在设计上假设只有一个核心执行内核核心代码，其他核心要么空闲，要么执行Privileged-1层服务。关键调度操作（如能力验证、逻辑核心分配）都在禁用中断的上下文中完成，不需要使用原子操作或内存屏障。这使HIC的调度器在x86、ARM和RISC-V上可以共享同一套实现，而不需要为每种架构编写特殊的屏障代码。

逻辑核心的跨核迁移涉及共享能力表的并发访问。HIC通过从核辅助处理能力管理，将能力表的更新操作串行化，避免了竞争条件。迁移过程对持有逻辑核心能力的服务透明——能力句柄不变，底层映射表被原子更新。HIC的AMP架构将从核专门用于处理能力管理，将能力验证、添加、撤销等操作的计算负载从Core-0主核卸载，同时保证了能力操作的原子性。从核可以并行处理多个能力操作，但每个能力表条目的更新是串行的，避免了传统多核调度器中复杂的锁竞争。

HIC在弱内存架构上的适应性。由于Core-0的核心调度路径不使用共享内存的并发访问，ARM和RISC-V的弱内存模型对HIC调度器的正确性影响有限。能力表的更新由从核串行化执行，自然提供了顺序一致性。HIC的无锁设计是规避架构差异的有效手段——与其编写复杂的跨架构屏障代码，不如从架构上消除并发访问的需求。

## 77.7 本章小结

x86-TSO的强内存模型使调度器的并发实现相对简单，LOCK前缀的原子操作提供了完整的顺序保证。ARMv8的弱内存模型需要显式的Load-Acquire/Store-Release屏障和DMB/DSB指令来保证正确的内存顺序。RISC-V RVWMO的极弱模型要求开发者深入理解FENCE指令和非多拷贝原子性。从x86迁移到弱内存架构时，原本正确的调度器代码可能暴露隐藏的并发错误，LKMM为在弱内存架构上编写正确并发代码提供了形式化框架。调度器中的无锁同步技术（RCU、无锁队列、任务窃取）高度依赖原子操作和内存屏障，在不同架构上有不同的实现策略。HIC通过无锁设计和Core-0单核执行从根本上规避了多核同步的复杂性，使其调度器在不同架构上具有统一的实现。理解架构差异对调度模型的影响，是编写可移植、高性能、正确的多核调度器的前提。
