# 第76章 构建与调试：Makefile、链接脚本、模拟器与硬件调试
内核开发不同于普通应用程序开发——内核代码运行在裸硬件之上，没有操作系统提供调试支持。当内核崩溃时，不能简单地打印栈回溯后退出；当内核行为异常时，不能直接附加调试器查看变量。内核的构建和调试是一整套独立的技术体系：构建系统负责将数万个源文件编译、链接成可引导的内核镜像，调试技术则负责在“没有操作系统”的环境中定位问题。本章讨论Kbuild构建系统、链接脚本、模拟器调试和硬件调试四个层面的核心技术。

76.1 Kbuild：Linux内核的构建系统

Linux内核不是简单的Makefile工程。Kbuild（Kernel Build）是专门为内核设计的分层递归构建系统。与普通Makefile不同，Kbuild必须处理数千个源文件、数百个配置选项、多种目标架构和复杂的依赖关系。

Kbuild的基本工作流程从配置开始。开发者运行make menuconfig、make xconfig等命令，通过Kconfig系统选择需要的功能。Kconfig是一套声明式的配置语言，每个功能模块在自己的目录下有Kconfig文件，定义该模块的配置选项、依赖关系和默认值。配置系统最终生成.config文件，保存所有配置选项的选择结果。

配置完成后，Kbuild开始编译。Kbuild的核心是递归Make结构：顶层Makefile读取.config，根据配置决定进入哪些子目录；每个子目录的Makefile或Kbuild文件定义该目录需要编译的目标——obj-y表示编译进内核（内置），obj-m表示编译为可加载模块（.ko），lib-y表示编译为静态库。每个子目录的Makefile并不直接调用编译器，而是定义需要编译的源文件列表，实际的编译规则由scripts/Makefile.build统一提供。这种设计避免了在每个子目录重复编写编译规则，也使交叉编译变得简单——只需在顶层指定ARCH和CROSS_COMPILE变量。

编译过程产生多个重要产物。vmlinux是未压缩的内核ELF文件，包含完整的调试符号，是调试和分析的核心文件。bzImage（x86）或Image（ARM）是经过压缩和封装的、可被引导加载程序直接加载的内核镜像。内核模块（.ko）是可重定位的ELF文件，在运行时由内核加载器动态链接到内核地址空间。

Kbuild还支持外部模块（out-of-tree module）编译，允许开发者在不重新编译整个内核的情况下独立编译驱动模块。外部模块只需要一个简单的Makefile，包含obj-m := mydriver.o和目标内核源码路径，就可以利用Kbuild的完整基础设施进行编译。

76.2 链接脚本：内存布局的施工蓝图

编译将源文件转换为目标文件（.o），但最终的内核镜像需要将所有目标文件合并成一个连续的内存映像。合并的方式由链接脚本（linker script）定义。链接脚本描述输入目标文件的各个节（section）如何合并、放置在输出文件的什么位置、以及运行时应该加载到内存的哪个地址。

链接脚本的核心命令是SECTIONS，它描述输出文件的内存布局。一个典型的链接脚本包含以下部分：.text节存放所有可执行代码；.rodata节存放只读数据；.data节存放已初始化的全局变量；.bss节存放未初始化的全局变量，在二进制文件中不占用空间，但加载时需要在内存中分配并清零。

Linux内核的链接脚本不是直接编写的，而是通过预处理生成。arch/$(ARCH)/kernel/vmlinux.lds.S是链接脚本模板，使用C预处理器宏和asm-generic/vmlinux.lds.h中定义的通用宏来生成最终的.lds文件。这种方式使不同架构可以共享通用的节布局规则，同时保留架构特定的调整能力。

链接脚本在内核启动中扮演关键角色。_stext、_etext、_sdata、_edata、__bss_start、__bss_stop等符号由链接脚本定义，在C代码中作为外部变量引用。启动汇编代码通过这些符号定位各段的边界，执行BSS清零、数据段复制等初始化操作。链接脚本还负责将.init.text节（内核初始化代码）放置在内存的特定位置，使内核在完成初始化后可以释放这部分内存。

76.3 模拟器调试：QEMU与GDB的联合调试

在真实硬件上调试内核非常困难——需要两台机器（调试主机和目标机）、串口或JTAG连接，且每次修改都需要烧录固件。QEMU模拟器提供了更便捷的替代方案。QEMU内置了GDB stub，允许GDB通过网络远程连接到运行中的虚拟机，像调试用户程序一样调试内核。

使用QEMU+GDB调试需要几个步骤。首先编译带调试信息的内核：在make menuconfig中启用CONFIG_DEBUG_INFO（包含调试符号）和CONFIG_GDB_SCRIPTS（提供GDB辅助脚本）。编译完成后得到未压缩的vmlinux（包含完整符号）和压缩的bzImage（用于启动）。其次制作根文件系统（initramfs），最简单的方案是使用BusyBox构建最小根文件系统，提供基本的shell和工具。

启动QEMU时添加-s参数（等价于-gdb tcp::1234）启用GDB stub，添加-S参数使CPU在启动时暂停，等待GDB连接。在另一个终端启动GDB，加载vmlinux符号文件，通过target remote :1234连接到QEMU，然后就可以像调试普通程序一样设置断点、单步执行、查看变量。内核源码树中的vmlinux-gdb.py脚本提供了一系列辅助命令，如lx-symbols（加载内核模块符号）、lx-current（获取当前进程）等。

模拟器调试的优势是无需硬件、可重现、支持快照和逆向执行，特别适合内核开发的初期调试和教学场景。

76.4 硬件调试：KGDB与kdb

当问题在真实硬件上出现且无法在QEMU中复现时，需要硬件调试手段。Linux内核提供了KGDB（Kernel GDB）——运行在内核中的GDB代理。KGDB允许GDB通过串口、网络或USB连接到运行中的内核，进行源码级调试。

启用KGDB需要在内核配置中打开CONFIG_KGDB，并选择KGDB I/O驱动（通常是CONFIG_KGDB_SERIAL_CONSOLE）。编译后，在目标机的内核命令行中添加kgdboc=ttyS0,115200（指定串口设备和波特率）和kgdbwait（使内核在启动早期等待调试器连接）。目标机启动后，在调试主机上运行gdb vmlinux，通过target remote /dev/ttyS0连接，即可开始调试。

KGDB支持断点、单步执行、查看内存和变量、获取调用栈等标准GDB功能。对于无法使用GDB的场景，内核还提供了kdb——一个内置的命令行调试器。kdb不需要外部GDB，可以直接在系统控制台上使用。通过SysRq+g组合键（需要开启CONFIG_MAGIC_SYSRQ）进入kdb shell，可以执行ps（查看进程）、bt（调用栈）、dmesg（日志）、md（内存显示）等命令。kdb适合快速诊断内核挂起或panic问题，不需要第二台机器。

KGDB与kdb可以同时启用。当GDB连接时，kdb的控制权让位给GDB；断开GDB后，kdb可以重新接管。这种组合为内核调试提供了从快速诊断到深度源码分析的全方位工具链。

76.5 分级隔离内核中的构建与调试

分级隔离内核（如HIC）的构建与调试设计遵循“最小化可信计算基”的原则。Core-0的代码量被严格控制在较小范围内，构建系统相应地比Linux Kbuild简化。配置选项被压缩到最少——不需要处理数千个Kconfig选项，只需要为目标硬件平台选择必要的功能。链接脚本也更直接，不需要复杂的预处理宏。

HIC构建系统将Core-0内核、Privileged-1层服务模块和硬件描述文件（platform.yaml）一起编译，生成单一内核镜像。在调试方面，Core-0支持QEMU模拟器调试，开发者可以在不依赖硬件的情况下验证Core-0的核心逻辑。对于Privileged-1层服务，HIC允许将服务运行在用户态模拟环境中进行调试，利用标准的用户态调试工具，而不必每次都在完整的内核环境中测试。由于Core-0无锁设计减少了并发不确定性，调试过程中更容易复现问题。能力系统的集中式管理也简化了调试信息收集——能力表是系统状态的集中式描述，开发者可以通过检查能力表快速了解系统资源分配状态，而不必在多个子系统中追踪分散的状态。

76.6 本章小结

内核的构建与调试是内核开发的基础设施。Kbuild作为Linux内核的构建系统，通过Kconfig、递归Make和统一定义的编译规则处理数万源文件的编译和链接。链接脚本（linker script）定义内存布局，通过SECTIONS命令将各目标文件的节合并到输出文件，并在链接过程中定义内核启动所需的符号。QEMU模拟器与GDB的联合调试为内核开发提供了便捷的软件调试环境，通过QEMU内置的GDB stub实现源码级调试。KGDB和kdb是真实硬件上的调试手段——KGDB作为内核中的GDB代理支持远程源码级调试，kdb提供内置命令行调试器用于快速诊断。分级隔离内核的构建与调试设计以最小化TCB为原则，简化构建流程，并利用能力系统的集中式状态管理简化调试信息收集。理解构建与调试工具链，是进行内核开发和问题定位的基本功。
